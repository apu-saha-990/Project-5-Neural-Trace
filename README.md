# Neural-Trace v2

A blockchain forensics tool that traces where money came from, monitors wallets for suspicious activity, and generates professional investigation reports.

---

## What Problem Does This Solve

When a cryptocurrency fraud happens, the stolen money doesn't stay in one place. It moves — sometimes through dozens of wallets, sometimes across multiple exchanges, sometimes through services designed to hide the trail. Investigators trying to follow that money manually would need to click through hundreds of wallet pages one by one, taking notes, building a picture by hand. By the time they finish, the money has moved again. This tool automates that process. You paste one wallet address and it traces automatically — building a full map of every wallet the money touched, how much moved, and where it ended up.

---

## Real-World Use

This tool was built as part of an active fraud investigation involving a cryptocurrency presale scheme. The investigation confirmed over 125 wallets connected to the fraud, traced funds exceeding $376 million USD, and identified deposit accounts at major regulated exchanges. Forensic reports generated by this tool have been prepared for submission to legal counsel in connection with a RICO case. The goal is to produce evidence that can support subpoenas to exchanges — forcing them to reveal the identity of whoever owns the accounts the stolen money landed in.

The reports this tool generates contain structured data — wallet tables, transaction records, spike alerts, and exchange identifications. The written narrative layer sits outside the tool for now: the exported JSON gets fed into a public AI, which reads the full dataset and produces a written investigative summary in plain English. That summary then gets compiled into the final document handed to legal counsel. Building a custom AI trained specifically on blockchain forensics and financial crime patterns — one that generates that narrative automatically as part of the export — is the next major phase of this project.

## How It Works

1. You open the tool in your browser — it asks for a username and password before letting you in
2. You paste a wallet address you want to investigate and give the case a project name
3. The tool follows the money outward — checking each destination wallet, then each of those, up to 7 layers deep
4. Every wallet gets classified automatically: a victim collection point, a passthrough, an exchange account, a mixer, and so on
5. A live map draws itself on screen as the trace runs — you can see the money flow happening in real time
6. When the trace is done, you click Export and get two files: a structured data file and a formatted Word document ready to hand to a lawyer
7. You can then switch to the Monitor tab to watch those same wallets for new activity — large transfers, suspicious patterns, wallets that were dormant for months before suddenly activating
8. The Origin Trace tab works backwards — for each wallet it finds where the money originally came from, including which exchange sent it and whether that exchange requires identity verification
9. Every time you start the tool, 38 automated checks run first to confirm everything is working correctly

## What's Built

**The investigation dashboard** — a single-page web interface where everything happens. Runs in your browser, talks to the local server, shows a live network graph as wallets are discovered. Built to feel like a real intelligence tool, not a spreadsheet.

**The forward tracer** — follows money outward from a seed wallet. At each hop it fetches transaction history, identifies wallet types, and builds the graph. Stops when it hits a known exchange, a mixer, or the maximum hop depth.

**The live graph** — draws itself while the trace runs. Wallets appear as hexagonal nodes. Lines between them show money flow, with colour and thickness showing how much moved. Exchange targets pulse red. You can click any node to highlight the full chain between it and the origin.

**The monitor engine** — takes a completed trace and re-checks every wallet in it against a chosen time window. Detects spikes (single transfers above $50,000), structuring (multiple similar-sized transfers designed to avoid detection), sudden activations (wallets dormant for 90+ days that suddenly move), and shared gas funders (a sign that wallets are controlled by the same operator).

**The origin tracer** — works backwards from each wallet to find where it was funded. Shows a summary table: wallet address, funding source, number of hops back, which exchange was hit, and whether that exchange requires identity verification.

**The report generator** — takes any completed run and produces two files: a structured JSON for further analysis, and a formatted Word document with all findings, tables, and narrative ready to share. Word format was chosen deliberately — legal teams can edit and annotate Word files in ways they can't with PDFs, which matters in practice.

**The wallet registry** — a lookup table of 76 known exchange addresses, mixers, and bridges. Every wallet found during a trace gets checked against this registry instantly.

**The login protection** — the tool requires a username and password before the browser will load anything. Credentials are set in a local configuration file and never hardcoded anywhere in the application.

**The test suite** — 38 automated tests that run every time you start the tool. They check that the analysis logic, exchange detection, and operator identification all work correctly before anything else runs.

## Bugs I Found

### Bug 1 — The monitor terminal flooding with raw data

The monitor would run, the terminal would fill up with output, and then at the end the screen turned into a wall of raw data that was completely unreadable.

When I looked at the server code, I found the monitor endpoint was actually spawning two separate processes for every run. The first one was supposed to stream live log lines to the browser. The second one was supposed to capture the final result. The problem was the first process had no way to separate its log output from its result — it was using a plain print statement with no marker, so the entire result — thousands of lines of JSON — was being treated as a log message and streamed directly into the terminal as plain text.

The second process was doing it correctly — it used a special marker in its output so the server could split the logs from the result. But by then the first process had already flooded the browser.

```javascript
// Before — two processes, first one dumps raw JSON into the terminal
runPython(['-c', `...print(json.dumps(result))`], onLog, onDone, onError);
const proc2 = ef(python, ['-c', `...sys.stdout.write('__RESULT__' + json.dumps(result))`]);

// After — one process, clean separation of logs and result
const proc2 = ef(python, ['-c', `...sys.stdout.write('__RESULT__' + json.dumps(result))`]);
```

The fix was removing the first process entirely. The second one was already doing everything correctly. One line deleted, problem gone. The lesson: when something looks like it's working but the output is wrong, the problem is usually earlier in the pipeline than you think. I was looking at the frontend rendering code for days before I found the duplicate process in the server.

### Bug 2 — The graph crashing mid-trace with "node not found"

During development I refactored how wallet data was stored internally to clean up the export format. After the refactor, the graph stopped loading mid-trace — throwing `node not found` errors in the log.

The problem wasn't obvious at first. The tracer stores wallets in a lookup table using lowercase addresses as keys — because Ethereum addresses are case-insensitive and mixing cases causes mismatches. The graph renderer needs each wallet to have an `id` field to link everything together. After the refactor, I set `id` to the original address exactly as returned by the API — which sometimes comes back with uppercase letters mixed in.

The visual graph uses a separate system to draw the connection lines between wallets. That system looks up wallets by their `id` to resolve which node is the source and which is the destination. When the `id` was mixed-case but the lookup table key was lowercase, it couldn't find the wallet and crashed.

```javascript
// Before — id was set to original address, case not guaranteed
id: n.addr  // could be "0x443fBE5c...047F"

// After — force lowercase to match the lookup table
id: (n.addr || '').toLowerCase()  // always "0x443fbe5c...047f"
```

The fix was one line but finding it took a while because the error message is generated deep inside the graph library — it doesn't tell you which field caused the mismatch. The lesson: when you have a lookup table with a specific key format, everything that references that table needs to use the exact same format. No exceptions.

I use AI (Claude) throughout development — as a learning tool, code reviewer, and debugging partner. Every terminal error went back to Claude. The decisions are mine: what to build, how to structure it, what the output should look like, what a forensic investigator actually needs in a report. I made those calls.

The architecture decisions — keeping the API key server-side, separating the graph renderer from the data pipeline, building the three analysis modes independently of each other, adding login protection so the tool can be demonstrated safely — those came from working through real problems. What happens when the API key is visible in browser source? What happens when graph coordinates end up in the exported data and pollute it? I ran into both and fixed them. The systems run. The tests pass. I can demo everything live.

## What I Learned

- Ethereum addresses look like plain text but they're case-insensitive — mixing cases silently breaks lookups in ways that are hard to spot
- Free public APIs have hard rate limits that affect investigation data quality — 500 transactions per call means high-volume wallets are always incomplete, and that has to be disclosed in any report used for legal purposes
- How to stream live data from a server to a browser without the page refreshing — and what breaks when you mix two different approaches to the same stream
- Keeping secrets server-side while serving a browser tool requires a specific pattern — the server injects the key into the page before sending it, so the browser never sees it in source
- Generating Word documents from code produces better results than PDF for legal work — legal teams can edit and annotate Word files, which matters in practice
- Structuring — breaking up large transfers into smaller ones to avoid detection — has specific mathematical patterns that can be caught automatically
- How to build a test suite that runs without needing a live API key, by replacing real data calls with controlled fake data in the tests
- The hardest part of building an investigation tool is deciding what information is actually useful versus what just looks impressive — I kept cutting things that added complexity without adding value to someone reading a report

## Running It

```bash
# Clone and enter the directory
git clone https://github.com/yourusername/neural-trace.git
cd neural-trace

# Copy environment file and fill in your credentials
cp .env.example .env
nano .env

# Run setup — installs dependencies, runs tests, starts the tool
chmod +x setup.sh
./setup.sh
```

## Environment Variables

```bash
ETHERSCAN_API_KEY=your_key_here   # Free key from etherscan.io/myapikey
NEURAL_TRACE_PORT=8080            # Port to run on (default: 8080)
NEURAL_TRACE_USER=admin           # Login username for browser access
NEURAL_TRACE_PASS=changeme        # Login password — change this before use
LOG_LEVEL=INFO                    # Logging verbosity
```

## Tech Stack

| What it does | Technology |
|---|---|
| Serves the web interface and handles all API requests | Node.js |
| Fetches wallet transaction data from the blockchain | Etherscan API v2 |
| Analyses transactions for spikes, structuring, and dormancy | Python 3 |
| Generates Word document reports | docx (Node.js library) |
| Draws the live wallet graph | D3.js |
| Streams live progress from server to browser | Server-Sent Events (SSE) |
| Stores credentials and API keys securely | Environment variables (.env) |
| Protects the interface with a login prompt | HTTP Basic Authentication |
| Runs automated checks on startup | pytest |
| Fetches live ETH price | CoinGecko API |
